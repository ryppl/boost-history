Introduction

The String Algorithm Library provides a generic implementation of various string-related algorithms which are missing in STL. It is an extension to the algorithms library of STL. Algorithms in this library include trimming, case conversion, predicates and find/replace functions. All of them come in different variants so it is easier to choose the best fit for a particular need.

The implementation is not restricted to work with a particular container ( like a std::basic_string ), rather it is as generic as possible. This generalization is not compromising the performance since algorithms are using container specific features when it means a performance gain.

The library also provides a good foundation for generic replace like transformations and searching. Core library algorithms are functor based to be easily customizable. Users can provide an implementation to fit their specific needs. An example of such a customization is regex part of the library ( see <boost/string_algo/regex.hpp> ) and RLE compression example ( see <libs/string_algo/example/rle_example.cpp> )

Reference

The library is organized into several parts and each part deals with a different set of problems:

  • Trim facility provides algorithms for removing trailing and leading spaces from sequences.
  • Case conversion facility contains sequence variants for std::toupper and std::tolower algorithms.

  • Predicates facility includes some popular string related predicates like starts_with, contains and etc.

  • Find facility provides a set of algorithms for searching for a substring within a sequence.

  • Replace/Erase facility contains various search and replace algorithms for sequences.

Every component is provided in separate header. The library contains two cumulative headers for convenience. <boost/string_algo.hpp> contain all components except regex related ones. <boost/string_algo_regex.hpp> includes also regex algorithms.

Build

Whole library is provided in headers. Regex variants of some algorithms, however, are dependant on the Regex++ library. All such algorithms are separated in <boost/string_algo/regex.hpp> . If this header is used, an application must be linked with the Regex++ library.

Portability

Library has been successfully compiled and tested with Microsoft Visual C++ 7.0 and GCC 3.2.

There are known limitation on platforms not supporting partial template specialization. Library depends on correctly implemented std::iterator_traits class. If a standard library provided with compiler is broken, the String Algorithm Library cannot function properly. Usually it implies, that primitive pointer iterators are not working.

Rationale

Copy vs. Mutable

Many algorithms in the library are performing a transformation of the input. Transformation can be done in-place, mutating the input sequence, or a copy of the transformed input can be created, leaving the input intact. None of these possibilities is superior to the other one and both have different advantages and disadvantages. For this reason, both are provided with the library.

Copy versions return a transformed input as a result. Mutable variants return a reference to the input. Thus both versions allow a simple chaining of transformations within one expression ( i.e. one can write trim_copy(toupper_copy(s)) as well as trim(toupper(s)) ).

Naming

Naming follows the conventions from the Standard C++ Library. If there is a copy and mutable version of the same algorithm,
the mutable version has no suffix and the copy version has suffix _copy.

Regex

Regular expressions are essential part of text processing. For this reason, the library provides also regex variants of algorithms. In some places they copy a functionality from the Regex++ library. The goal was to provide complete unified interface for all relevant algorithms in the library.
