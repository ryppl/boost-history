Boost.IndexedSet Future work

A number of new functionalities are considered for inclusion into future releases of Boost.IndexedSet. Some of them depend on the potential for extensibility of the library, which has been a guiding principle driving the current internal design of indexed_set.

Hashed indices

Several STL implementations feature hashed sets as a natural counterpart to std::set and std::multiset. indexed_set can also benefit from the inclusion of hashed indices. As the exact details of the interfaces of hashed sets differ among library vendors, a good starting point seems Matt Austern's paper A Proposal to Add Hash Tables to the Standard Library (revision 4), which has been submitted for acceptance into the next revision of the C++ standard.

Sentries

A sentry index can be thought of as a void index whose only purpose is to receive events from the enclosing indexed_set (insertion, erasing, modifying of elements) and signal them to an external entity --for instance, by means of the Boost.Signals library. This special type of index can have applications for:

  1. Logging,
  2. interfacing to GUI-based applications,
  3. synchronization between separate data structures.

User-defined indices

The mechanisms by which Boost.IndexedSet orchestrates the operations of the indices held by an indexed_set are simple enough to make them worth documenting so that the (bold) user can write implementations for her own indices.

Indexed maps

indexed_set is rich enough to provide the basis for implementation of indexed maps, i.e. maps which can be looked upon several different keys. The motivation for having such a container is mainly aesthetic convenience, since it would not provide any additional feature to similar constructs based directly on indexed_set.

The main challenge in writing an indexed map lies in the design of a reasonable interface that resembles that of std::map as much as possible. There seem to be fundamental difficulties in extending the syntax of a std::map to multiple keys. For one example, consider the situation: 

indexed_map<int,string,double> m;
// keys are int and string, double is the mapped to value

...

cout<<m[0]<<endl;      // OK
cout<<m["zero"]<<endl; // OK
m[1]=1.0;              // !!

In the last sentence of the example, the user has no way of providing the string key mapping to the same value as m[1]. This and similar problems have to be devoted a careful study when designing the interface of a potential indexed map.

Serialization support

Once Robert Ramey's serialization library gets accepted into Boost, support for archiving/retrieving indexed_sets should be added.

Mojoing indexed_set

Andrei Alexandrescu introduced a technique for simulating move constructors called Mojo (see his article in C/C++ User Journal "Generic<Programming>: Move Constructors".) Mojo alleviates the computational load involved in the creation and copying of temporary objects, specially for heavy classes as indexed_sets are.

A mojoed indexed_set is seen as particularly useful when used as an internal building block in other libraries (vg. relational database frameworks), enabling the efficient development of functions returning indexed_sets. Without mojoing, this scheme is impractical and less elegant syntaxes should be resorted to.
