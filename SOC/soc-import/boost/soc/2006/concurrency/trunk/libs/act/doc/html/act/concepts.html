Consumer

A Consumer is an object which takes in data of a single type one instance at a time and processes it in an implementation defined-manner either sychnronously or asynchronously with respect to the thread it is interfaced with. In Boost.Act, models of this concept may be found as members of models of the Consumer Policy concept.

In order to be considered a valid Consumer, given the variable definitions below, the following expressions must be well-formed and have the indicated effects.

Table 2. Worker Descriptive Variable Definitions

Variable Definition
C A Consumer type.
c Value of type C&.
A An policy type which may be used in conjunction with C. Note: Such policy types are implementation-specified with respect to models of the Consumer concept. --end note
a Value of type A&.
d Value of type C::value_type const&.

Table 3. Worker Requirements

Expression Return Type Pre/Post-Condition

C::value_type   The type of data which is processed.
c.consume(d) (unused) Processes a copy of d. Note: Processing takes place in an implementation-defined thread of execution. --end note

Consumer Policy

A Consumer Policy is a type whose instances are used to create Consumer objects. These Consumer Policys generally act as policies to higher-level types. In Boost.Act, such allocator types are used, most notably, as policies during active-qualification.

In order to be considered a valid Consumer Policy, given the variable definitions below, the following expressions must be well-formed and have the indicated effects.

Table 4. Worker Policy Descriptive Variable Definitions

Variable Definition
A A Consumer Policy type.
a Value of type A&.
T The type of objects to be processed.
t Value of type T&.
tc Value of type T const&.
P A unary function object which may take t as an argument.
p Value of type P const&.
sc A value of type A::scoped_consumer&
ssc A value of type A::safe_scoped_consumer& with the corresponding object constructed with a as an argument.

Expression Requirements

Table 5. Worker Policy Requirements

Expression Return Type Pre/Post-Condition
A::scoped_consumer<T,P>   A Consumer type which guarantees all processing is completed prior to the object being destroyed.
A::safe_scoped_consumer<T,P>   A Consumer type which guarantees all processing is completed prior to the object being destroyed and which may be interfaced with by any number of threads of execution.
A()   Creates a default instance. A destructor is assumed.

Interlocked Object

An Interlocked Object is an object which supports thread-safe interlocked operations and which are often used in the implementation of lock-free code. On systems which support interlocked operations, all non-member and pointer types and some arithmetic types may be compliant Interlocked Objects as well as some library-defined or user-defined types given that they support the required operations. For details on how to select and take advantage of such types, see the _interlocked_.

In order to be considered a valid Interlocked Object, given the variable definitions below, the following expressions must be well-formed and have the indicated effects.

Table 6. Interlocked Object Descriptive Variable Definitions

Variable Definition
I An Interlocked Object type.
IV I volatile.
C1 A type which is convertible to IV.
C2 A type which is convertible to IV.
iv Value of type I volatile&.
c1 Value of type C1 const&.
c2 Value of type C2 const&.

Expression Requirements
Note

In the following requirements, certain operations are listed as having acquire memory semantics, release memory semantics, or a full memory barrier. Operations with acquire memory semantics imply that the memory operation being performed by the current thread will be visible before any other memory operations are attempted. Operations with release memory semantics imply that the memory operation being performed by the current thread will be visible after all other memory operations have been completed. Operations with a full memory barrier imply that the memory operation being performed by the current thread enforces a strict ordering with both acqure memory semantics and release memory semantics. These specifications are merely a minimum requirement and therefore on certain systems, such operations which require acquire memory semantics or release memory semantics may be implemented with a full memory barrier. For brevity, functions and templates in the namespace ::boost::act::interlocked are prefixed simply with interlocked::.

Table 7. Interlocked Object Requirements

Expression Return Type Pre/Post-Condition
I()   Constructs an instance of I with an unspecified initial value.
I i = c1   Constructs an instance of I with c1 as an initial value.
iv = c1 I& Stores the value of c1 to object i. Returns a reference to iv. Note: This is not an interlocked operation and does not imply a memory barrier of any kind. --end note
interlocked::assign( iv, c1 ) I Stores the value of c1 to object iv with a full memory barrier. Returns the value of iv prior to the operation being performed.
interlocked::assign_acquire( iv, c1 ) I Stores the value of c1 to object iv with acquire memory semantics. Returns the value of iv prior to the operation being performed.
interlocked::assign_if_was( iv, c1, c2 ) I Stores the value of c1 to object iv iff iv == c2 prior to the operation with a full memory barrier. Returns the value of iv prior to the operation being performed. Note: This is the same as the more commonly known "compare and swap" operation. --end note
interlocked::assign_if_was_acquire( iv, c1, c2 ) I Stores the value of c1 to object iv iff iv == c2 prior to the operation with aquire memory semantics. Returns the value of iv prior to the operation being performed.
interlocked::assign_if_was_release( iv, c1, c2 ) I Stores the value of c1 to object iv iff iv == c2 prior to the operation with release memory semantics. Returns the value of iv prior to the operation being performed.
interlocked::volatile_get( iv ) I Returns the value of iv. Note: Used for retrieving the value of a potentially volatile-unqualified type as though it were volatile-qualified. --end note

Worker

A Worker is an object which represents a function executing either sychnronously or asynchronously with respect to its thread of construction depending on the corresponding model and implementation. In Boost.Act, models of this concept may be found as members of models of the __workerallocator_ concept.

In order to be considered a valid Worker, given the variable definitions below, the following expressions must be well-formed and have the indicated effects.

Table 8. Worker Descriptive Variable Definitions

Variable Definition
W A Worker type.
cw Value of type W const&.
A An allocator type which may be used in conjunction with Worker w. Note: Such allocator types are implementation-specified with respect to models of the Worker concept. --end note
a Value of type A&.
f A nullary function object.

Expression Requirements

Table 9. Worker Requirements

Expression Return Type Pre/Post-Condition
W(a,f)   Creates an instance which executes f in an implementation-specified thread of execution. A destructor is assumed.
cw.join() (unused) Post: The function passed in during construction has completed its execution. Note: This function may be called multiple times without error. No matter how many times the function is called, the corresponding function will have only been run once. --end note

Worker Allocator

A Worker Allocator is a type whose instances are used to create Worker objects which execute user-specified functions. These Worker Allocators generally act as policies to higher-level types.

In order to be considered a valid Worker Allocator, given the variable definitions below, the following expressions must be well-formed and have the indicated effects.

Table 10. Worker Allocator Descriptive Variable Definitions

Variable Definition
A A Worker Allocator type.
a, a1, a2 Values of type A& where a1 == a.
uw A value of type A::unmanged_worker& with the corresponding object constructed with a1 as an argument.
suw A value of type A::safe_unmanaged_worker& with the corresponding object constructed with a1 as an argument.
f A nullary function object.

Table 11. Worker Allocator Requirements

Expression Return Type Pre/Post-Condition
A::unmanaged_worker   A Worker type which may be joined from a single thread and which must be joined prior to the parent process's completion to be well-formed.
A::safe_unmanaged_worker   A Worker type which may be joined from any number of threads and which must be joined prior to the parent process's completion to be well-formed.
a.spawn_wild_worker(f) (unused) f is executed in an implementation-defined thread. Note: f is guaranteed to complete its execution prior to a natural closing of the corresponding process. --end note
a1 == a2 bool Returns true iff workers allocated from each can be joined via the other.
a1 != a2 bool Same as !(a1 == a2).
A()   Creates a default instance. A destructor is assumed.

Threaded Worker

A Threaded Worker is type which may be used to execute user-specified functions in a separate thread from that of its of construction. In Boost.Act, models of this concept may be found as members of models of the __threaded_workerallocator_ concept.

Refinement of
  • Worker

As a refinement of the Worker concept, a Threaded Worker must meet all of the requirements of the Worker concept along with the aformentioned restrictions. There are no additional interface requirements for a type to be considered a valid Threaded Worker.

Threaded Worker Policy

A Threaded Worker Policy is a type which may be used to create models of Threaded Worker concept. In Boost.Act, Threaded Worker Policys are used as policies to the __concurrent_consumerpolicy model.

Refinement of
  • Worker Policy

As a refinement of the Worker Policy concept, a Threaded Worker Policy must meet all of the requirements of the Worker Policy concept along with the aformentioned restrictions. There are no additional interface requirements for a type to be considered a valid Threaded Worker Policy.
