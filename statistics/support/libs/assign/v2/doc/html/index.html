Introduction

This library offers a uniform interface for initializing or assigning containers, and working with collections of references. The relevant directory structure is rooted boost/assign/v2. Each directory in this structure contains an .hpp file by the same name which includes all its relevant files. We assume, throughout this documentation, that 

#include <boost/assert.hpp> 
#include <boost/range.hpp>
#include <boost/assign/v2/v2.hpp>
using namespace boost::assign::v2;

precedes all C++ statements.

Important

Elements are passed by reference. In the example involving a tuple, it may have, for example, an lvalue and a const reference as its elements.

To pass a Range, from, to an arbitrary container, to, 

put_range( from, to );

To assign a Range to a container, 

to = convert<To>( from );

Anonymous container (of values)

An anonymous container is created by using the same interface as that of put and has most of the functionality of deque_ plus the conversion capability (See ContainerConvertible). For assignment, 

to = anon( a )( b )( c );

To copy-construct to, 

To to = anon( a )( b )( c ).convert<To>();

To be able to use operator%, 

( anon<t_>( _nil ) % (_repeat = n ) )( a )( b )( c );

Anonymous container of references

The array returned by 

using namespace ref;
anon<ref-assign-tag>( a )( b )( c );

dereferences to a, b, and c, respectively. Clearly only unary function calls are allowed, but there is a csv counterpart as well (see below). The template argument ref-assign-tag specifies reference assign semantics. Rather than specifying ref-assign-tag, it is equivalent to use a dedicated namespace. For copy semantics, 

using namespace ref;
using namespace assign_copy;
boost::copy( v , boost::begin( anon( a )( b )( c ) ) );
BOOST_ASSERT( a == v[0] );
BOOST_ASSERT( c == v[2] );

For rebind semantics, 

#include <boost/typeof/typeof.hpp> 
using namespace ref;
using namespace assign_rebind;
BOOST_AUTO( tmp, csv_anon( a , b ) );
tmp.assign( c );
BOOST_ASSERT( &tmp[ 0 ] == &c );
BOOST_ASSERT( &tmp[ 1 ] == &c );

Chaining ranges

To assign from chained ranges: 

to = (r1 && r2 && r3).convert<To>();

Chaining is elementwise-lvalue preserving: 

#include <boost/typeof/typeof.hpp> 
using namespace ref;
using namespace assign_copy;
BOOST_AUTO( tmp, anon( a )( b ) );
copy( v , boost::begin( tmp && anon( c ) ) );
BOOST_ASSERT( a == v[0] );
BOOST_ASSERT( b == v[1] );
BOOST_ASSERT( c == v[2] );

The fact that anon( c ), above, returns an rvalue should underscore the significance of elementwise in our previous statement. See RangeLvalue. Continuing our example, 

tmp.convert<v_>();

Library interface and design

Functions are the primary interface by which the user interacts with the library. From an implementation standpoint, however, it is the object that they return that are responsible for performing the desired task. This relationship is formalized as follows:

Interface

Implementation 

fun( a )( b )( c );


result_of::fun_nil<t_>::type object = fun<t_>( _nil ); 
object( a )( b )( c );

To avoid passing to functions many, in fact any, parameters that would confuse the user and cause the developer a headache, the library usually deduces the parameters from its input using traits. Considering, for example, the put() functionality, the library, by default,

  • For creating each element, calls a functor, constructor<t>(), overloaded on the number and lvalue/const-ness of its arguments (see K-Ary).
  • A traits decides which modifier (such as push_back()) to use given certain properties of the container. For example, if the container has push() it takes precedence over push_back().

For example, the side effect of 

put( map )( key1, a )( key2, b )( key3, c )

is identical to that of 

map.insert( make_pair( key1, a ) ); 
map.insert( make_pair( key2, b ) ); 
map.insert( make_pair( key3, c ) );

The deduced parameters that we have just discussed can be selectively overriden with operator% taking as its first argument the result of the function, and a keyword. A keyword is a const object defined in an anonymous namespace whose identifier has prefix _. Here's the generic form 

object % _keyword;

or 

object % (_keyword = p);

Operators are usually composable e.g. 

object % _keyword1 % (_keyword2 = p);

In each case, the operation may return a reference to the same object, but with a different state, or a new object of a different type altogether, thereby modifying the semantics of subequent calls to the object. For example, 

(put( map ) % _incr_lookup)( a )( b )( c )

is equivalent to 

++ map [ a ]; ++ map [ b ];  ++ map [ c ];

Portability

Methodology

For each of a set of containers (see below), ensure

  • At compile time, that the method deduced for modifying the container agrees with one that is explicitly specified. For example, it verifies that push() is the method used for std::queue<>.
  • At runtime time, that the container returned by 
    convert<To>( from );
    has the same elements as that of from, where To is the type of the container being tested.

Change log

Upgrading from Boost. v1.4x
  • This upgrade to Boost.Assign v2 represents a complete overhaul and is completely independent from the previous version. The new features are in boost/assign/v2, and accordingly, a new namespace was created: 

using namespace boost::assign::v2;

  • The latest version prior to Boost.Assign v2 will continue to be supported in Boost for backward compatibility purposes, and also because Boost.Assign v2 still needs to be developed in some areas such as pointers.
  • The change to Boost.Assign v2 comes about with the proposal for the addition of two new features, namely chaining ranges and (more advanced) anonymous containers of references. However, it came to light during the mini-review, that the new and existing features seemed redundant or lacked homogeneity in their interface and therefore called for a little more than a tune up.

Support

For questions or feature request, post to the Boost Users Mailing List, but also make sure to reach TO and ER by CCing them.

History and acknowledgement

The idea for an assignment/initialization library is not new. The functionality of this library resembles Leor Zolman's STL Container Initialization Library a great deal, but it does not rely on string parsing to achieve its goals.
