Introduction

This library offers 3 families of interfaces for initializing or assigning containers. One, which is uniform across container types, helps putting elements in the container, one by one, or from a range, possibly after some transformation. Another brings a collection of references under an array-like interface. This can be useful for applying algorithms to these collections, or as intermediate step in modifying the content a container. The third option allows for chaining ranges to form a new range that can be converted to an arbitrary container.

The relevant headers are in the directory structure rooted at boost/assign/v2. Each directory contains an .hpp file by the same name which includes all the files that are relevant to the user. We assume, throughout this documentation, that 

#include <boost/assert.hpp>
#include <boost/range.hpp>
#include <boost/assign/v2/v2.hpp>
using namespace boost::assign::v2;

precedes all C++ statements.

Important

Elements are passed by reference. In the example involving a tuple, it may have, for example, an lvalue and a const reference as its elements.

To pass a Range, from, to an arbitrary container, to, 

put_range( from, to );

To assign a Range to a container, 

to = convert<To>( from );

Each of these two forms have an alternative adaptor syntax which uses the pipe a.k.a the bitwise-or operator. The advantage of this syntax is its composition property.

Anonymous container (of values)

An anonymous container is created by using the same interface as that of put and has most of the functionality of deque_ plus the conversion capability (See ContainerConvertible). For assignment, 

to = anon( a )( b )( c );

To copy-construct to, 

To to = anon( a )( b )( c ).convert<To>();

Creating an empty container can be useful for applying operator% before any argument is passed: 

( anon<t_>( _nil ) % (_repeat = n ) )( a )( b )( c );

Anonymous container of references

The array returned by 

using namespace ref;
anon<ref-assign-tag>( a )( b )( c );

dereferences to a, b, and c, respectively. Clearly only unary function calls are allowed, and there is a csv counterpart as well (see below). The template argument ref-assign-tag specifies reference assign semantics, but it can ommitted by using a corresponding namespace. For copy semantics, 

using namespace ref;
using namespace assign_copy;
boost::copy( v , boost::begin( anon( a )( b )( c ) ) );
BOOST_ASSERT( a == v[0] );
BOOST_ASSERT( c == v[2] );

For rebind semantics, 

#include <boost/typeof/typeof.hpp>
using namespace ref;
using namespace assign_rebind;
BOOST_AUTO( tmp, csv_anon( a , b ) );
tmp.assign( c );
BOOST_ASSERT( &tmp[ 0 ] == &c );
BOOST_ASSERT( &tmp[ 1 ] == &c );

Chaining ranges

To assign from chained ranges: 

to = (r1 && r2 && r3);

To copy-construct from chained ranges: 

To to = (r1 && r2 && r3).convert<To>();

Chaining is elementwise-lvalue preserving: 

#include <boost/typeof/typeof.hpp>
using namespace ref;
using namespace assign_copy;
BOOST_AUTO( tmp, anon( a )( b ) );
copy( v , boost::begin( tmp && anon( c ) ) );
BOOST_ASSERT( a == v[0] );
BOOST_ASSERT( b == v[1] );
BOOST_ASSERT( c == v[2] );

The fact that anon( c ), above, returns an rvalue should underscore the significance of elementwise in our previous statement. See RangeLValue.

Library interface and design

The function--object relationship

Functions are the primary interface by which the user interacts with the library. From an implementation standpoint, however, it is the object that they return that are responsible for performing the desired task:

Interface

Implementation 

fun( a )( b )( c );


result_of::fun_nil<t_>::type object = fun<t_>( _nil ); 
object( a )( b )( c );

where t_ is deduced from the first argument, a. This representation is that which underpins anon() and ref::anon(). In the case of 

put( v )( a )( b )( c )

the value type, t_, is determined from the type of the container, v.

Default behavior for putting elements in a container

To avoid passing to a function many, in fact any, parameters that would confuse the user and cause the developer a headache, the library usually deduces the parameters from its input using traits. Considering the put() function above, the library, by default,

  • Creates each element by calling functor constructor<t>(). The latter is overloaded on the number and lvalue/const-ness of its arguments (see K-Ary), so that, 
    put( v )( a )()( c[0], c[1] )
    is equivalent to 
    put( v )( a )( t_() )( t_( c[0], c[1] ) )
  • Calls a particular container modifier, depending on the type of v. For example, whether v.push() is valid is detected at compile time, and if it is, is accepted as the modifier. If, however, the container models AssociativeContainer, insertion is called for, etc. For example, the side effect of 
    put( map )( key1, a )( key2, b )( key3, c )
    is equivalent to that of 
    map.insert( make_pair( key1, a ) ); 
map.insert( make_pair( key2, b ) ); 
map.insert( make_pair( key3, c ) );
Selectively overriding the default state or semantics of an object

The deduced parameters that we have just discussed can be selectively overriden with operator% taking as its first argument the result of the function, and a keyword. A keyword is a const object defined in an anonymous namespace whose identifier has prefix _. Here's the generic form 

object % _keyword;

or 

object % (_keyword = p);

Operators are usually composable e.g. 

object % _keyword1 % (_keyword2 = p);

In each case, the operation may return a reference to the same object, but with a different state, or a new object of a different type altogether, thereby modifying the semantics of subsequent calls to the object. For example, 

(put( map ) % _incr_lookup)( a )( b )( c )

is equivalent to 

++ map [ a ]; ++ map [ b ];  ++ map [ c ];

Portability

Methodology

For each of a set of containers (see below), ensure

  • At compile time, that the method deduced for putting elements into a container agrees with one that is explicitly specified. For example, it is asserted that push() is the method used for std::queue<>.
  • At runtime time, each of the elements of the container created as follows: 
    #include <boost/assign/v2/detail/checking/constants.hpp>
v_ v;
{
	using namespace checking::constants;
	put( v )( a )( b )( c )( d )( e )( f )( g )( h );
}
    are checked against a, b, c, d, e, f, g, and h, respectively, and that v has a size 8.

Change log

Upgrading from Boost. v1.4x
  • This upgrade to Boost.Assign v2 represents a complete overhaul and is completely independent from the previous version. The new features are in boost/assign/v2, and accordingly, a new namespace was created: 

using namespace boost::assign::v2;

  • The latest version prior to Boost.Assign v2 will continue to be supported in Boost for backward compatibility purposes, and also because Boost.Assign v2 still needs to be developed in some areas such as pointers.
  • The change to Boost.Assign v2 comes about with the proposal for the addition of two new features, namely chaining ranges and (more advanced) anonymous containers of references. However, it came to light during the mini-review, that the new and existing features seemed redundant or lacked homogeneity in their interface and therefore called for a little more than a tune up.

History and acknowledgement

The idea for an assignment/initialization library is not new. The functionality of this library resembles Leor Zolman's STL Container Initialization Library a great deal, but it does not rely on string parsing to achieve its goals.
