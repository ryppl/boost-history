Introduction

The configuration options shown in Configuring Boost.Persistent can not only be used to change template arguments of existing types, but also to replace types with own types that implement the same concept.

For example, instead of boost::persistent::multiversion_object_resource any other type implementing the ResourceManager concept can be used.

Concepts

The following concepts can be implemented to extend Boost.Persistent:

Table 1.2. Concepts for extending Boost.Persistent

Concept

Usage

TransactionManager

By implementing a TransactionManager, the start, commit, rollback of global transactions and how they are bound to threads can be controlled. A TransactionManager also decides which ResourceManagers are used for a certain Service.

ResourceManager

A ResourceManager offers one or more Services to the user interface. For example a ResourceManager that offers an ObjectStorageService and that is chosen by the TransactionManager to be used for the storage of an object is called by locators (the user interface) to access the object. ResourceManagers have control over local transactions.

An example of a ResourceManager implementation is one that stores objects in an external database that is able to perform ACID transactions on its own, so requests to the ResourceManager by the TransactionManager to start a local transaction can be forwarded to that external database.

AtomicStorageEngine

An AtomicStorageEngine is typically used by a ResourceManager offering a ObjectStorageService. It also implements functions to store objects and local transactions, but transactions of an AtomicStorageEngine are much simpler: The only guarantee is that changes to objects are written atomically, so if a local transaction of an AtomicStorageEngine is committed it must be able to recover that state on restart. No automatic rollbacks are necessary and no other guarantees like isolation or concurrency.

For example, an external database that is able to recover from system failures but does not offer the necessary isolation guarantees can be used as a storage backend by implementing this concept.

StorageEngine

A StorageEngine must be able to store objects, without knowing about transactions. The only requirement is that writes must be repeatable: If writing to the StorageEngine was aborted and the application is restarted, the same calls using the same arguments must yield the same result.

redo_storage can be used to turn a StorageEngine into an AtomicStorageEngine.

StorageEngine can be implemented to store objects in files, in external databases not aware of transactions, etc.

DataBlock

A DataBlock is a very simple concept that offers random access to a stream of bytes. It can be used by block_storage to create a StorageEngine.

Implementations of DataBlock include files, files mapped to memory, other regions in memory, etc.


Services

A ResourceManager implements one or more services that can be used by the user interface. It is queried (statically) by the TransactionManager which services it offsers.

The services that are currently defined by Boost.Persistent are:

Table 1.3. Service concepts

Concept

Usage

ObjectStorageService

A ResourceManager offering ObjectStorageService must implement several functions providing access to persistent objects.

SharedObjectStorageService

Functions providing access to shared objects, to be used for example by a shared_loc. Can be implemented e.g. by reference counting.

ObjectPinningService

Functions that provide the ability to associate an object with a type, and later retrieve the same object by that type.
