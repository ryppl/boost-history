Upgrading from Boost v1.4x : Boost.Assign 2.0

Upgrading from Boost v1.4x : Boost.Assign 2.0

Library Boost.Assign 2.0

  • is defined in namespace boost::assign::v2 and therefore does not affect the prior version, Boost.Assign (1.0). See section Organization.
  • is a follow up to a mini-review in which only a subset of the present upgrade was considered.
Ref & Chain

The following changes can be traced to the mini-review:

  • The pair of functions in Boost.Assign (1.0), ref_list_of and cref_list_of, are replaced by a single function, ref::array, that resolves lvalue/const-ness and the number of arguments automatically (thus eliminating the size template parameter).
  • A dual function, ref::csv_array, such that ref::csv_array(x, y, z) is equivalent to ref::array( x )( y )( z )
  • A range adaptor for chaining ranges, with special consideration for reference-arrays. See the relevant section.
Shared design

This library addresses some demands that surfaced during the mini-review:

  • Generalizes the above functor/csv duality throughout the library.
  • Supports C++0x features. Functions are typically variadic, declaring arguments as Args&&... and, where applicable, forward to an implementation using std::forward<Args>. Under C++03, these aspects are emulated using overlads (Solution #3 of the argument forwarding problem.) within some limits that can be configured.
  • Removes the ambiguity, as to whether the creation of an empty array or a default constructed element is triggered by a nullary call to a container generator. See here.
Interpreter

Library Boost.Assign (1.0)'s list_of and list-inserters are replaced by deque and put, respectively. In this new framework, the two functions share most of their functionality through a crtp class, interpreter_crtp[9]. This new design conforms to the interface described under Shared design and is characterized by these features:

  • A syntax that is uniform across container categories
  • Options for overriding semantics. Options are customizable.
  • Operator| for on the fly modification (piping)

Some navigational pointers:

  • The introduction's example illustrates the first and last feature, above, and some aspects of the shared design.
  • Section Performance contains a speed comparison of expressions under Boost.Assign (1.0) and their nearest analogue under Boost.Assign 2.0
By-product

A by-product of the interpreter framework is a tool for conversion. Whereas in Boost.Assign (1.0) it is embedded in an object, such as that returned by list_of, in 2.0 it is free functions. This sacrifice in convenience must be balanced against safety and code decoupling. Some features were left out altogether, such as comparison operators, so as to avoid redundancy with range utilities.


[9] The word interpreter designates the same thing as proxy object in Boost.Assign (1.0).
